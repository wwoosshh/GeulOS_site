# 구조별 분류 (커널 설계)

커널은 OS의 핵심. 설계 철학에 따라 성능·안전성·복잡도가 갈린다.

## 모놀리식 커널 (Monolithic Kernel)

모든 OS 서비스(드라이버·파일 시스템·네트워크 스택·스케줄러)가 *단일 주소 공간*의 커널
모드에서 동작. 빠른 성능 + 직접 하드웨어 접근. 단점: 한 부분 버그가 전체 영향.

- **Linux** (커널 모듈로 부분 확장 가능)
- **FreeBSD / OpenBSD / NetBSD**
- **MS-DOS**
- **IBM AIX / HP-UX / Solaris**

## 마이크로커널 (Microkernel)

커널은 *최소* 기능만 (IPC, 기본 스케줄링, 메모리 관리). 드라이버·파일 시스템 등은
*사용자 공간* 서버 프로세스로. 모듈성·내결함성 우수. 단점: IPC 오버헤드.

- **MINIX 3** (Tanenbaum 교수, 교육용에서 자체 사용 OS로)
- **QNX Neutrino** (BlackBerry, 차량)
- **GNU Hurd** (GNU 프로젝트)
- **L4 계열** (L4Ka::Pistachio, Fiasco.OC)
- **seL4** — *형식 검증된* 마이크로커널 (수학적으로 무결성 증명)

## 하이브리드 커널 (Hybrid Kernel)

모놀리식의 성능 + 마이크로커널의 모듈성 절충. 핵심 서비스는 커널 공간, 나머지는
사용자 공간.

- **Windows NT** (3.1 ~ 11)
- **macOS / iOS** (XNU 커널 — Mach 마이크로커널 + BSD 모놀리식 결합)
- **BeOS / Haiku**

## 엑소커널 (Exokernel)

커널은 *하드웨어 보호와 자원 분배*만 담당. 추상화는 사용자 라이브러리에 위임. 실험적.

- **MIT ExOS**
- **Xen** (하이퍼바이저 — 엑소커널 원리 활용)
- **Barrelfish** (멀티커널)

## 나노커널 / 유니커널 (Nanokernel / Unikernel)

커널을 *극도로 최소화*해 인터럽트 컨트롤러·타이머조차 드라이버에 위임. 또는 *한 응용*
전용으로 커널+앱을 단일 이미지로 컴파일.

- **Mac OS 8.6+** (나노커널 설계 일부 도입)
- **MirageOS** (유니커널, OCaml)
- **IncludeOS** (유니커널, C++)
- **HermitCore**
